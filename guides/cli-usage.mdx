title: "CLI usage"
description: "Use the aomi CLI to chat with your App locally, validate a preamble or new tool, switch models, simulate a transaction, debug a tool call, and inspect or resume a session."
---

The `aomi` CLI lets you drive your App from the terminal, with no frontend to spin up. It talks to the same backend your Widget would. This page is a tour built around tasks. For the full list of commands and flags, see the [CLI reference](/reference/cli).

The CLI is not a process that keeps running. Each command runs and exits. Conversation history lives on the backend, and a small amount of local state (which session is active, your wallet address, pending transactions) is saved under `~/.aomi` between runs.

## Install

```bash
npm install -g @aomi-labs/client
aomi --help
```

You can also run any command without installing, using `npx @aomi-labs/client ...`. The examples below assume a global install, so the command is just `aomi`.

If your App is private, pass your API key with `--api-key` or set `AOMI_API_KEY` in your environment. There is no command that stores an API key for you.

<Note>
  The deployed unit you talk to is an **App**. Select which App you target with the `--app` flag (or the `AOMI_APP` environment variable). The default App is `default`.
</Note>

## Chat with your App locally

Use this to sanity check a preamble or a new tool before you deploy.

<Steps>
  <Step title="See which Apps you can target">
    ```bash
    aomi app list
    aomi app current
    ```
  </Step>
  <Step title="Start the interactive REPL">
    Run `aomi` with no arguments to open a chat loop:

    ```bash
    aomi --app mycoindex
    ```

    Type a message, press Enter, and watch the reply. Inside the REPL you have a few slash commands: `/app <name>` switches the active App, `/model <rig>` switches the model, `/key` manages your own provider key, and `/heap` prints help. Type `:exit` to quit.
  </Step>
  <Step title="Or send a single prompt and exit">
    ```bash
    aomi --prompt "what is the price of ETH?" --app mycoindex
    ```

    There is also an explicit `chat` subcommand if you prefer to pass the message as a positional argument:

    ```bash
    aomi chat "what is the price of ETH?" --app mycoindex
    ```
  </Step>
</Steps>

<Note>
  The REPL needs a real terminal. In a script or CI, use `--prompt` or `aomi chat "..."` instead.
</Note>

## Validate a preamble or new tool before deploying

Iterate on your system prompt or a freshly added tool without redeploying. Send the same prompt after each change and read the reply.

<Steps>
  <Step title="Send the prompt that should exercise the change">
    ```bash
    aomi chat "rebalance my index to 60 percent ETH" --app mycoindex
    ```
  </Step>
  <Step title="Watch the tool calls inline">
    Add `--verbose` (or `-v`) to see which tools the model called and what they returned:

    ```bash
    aomi chat "rebalance my index to 60 percent ETH" --verbose
    ```

    This is the fastest way to confirm your preamble change actually steers the model toward the right tool.
  </Step>
  <Step title="Read back the full exchange">
    ```bash
    aomi session log
    ```

    This prints the whole conversation, including tool results, so you can compare behavior across edits.
  </Step>
</Steps>

<Warning>
  The CLI does not load a preamble from a local file. It always talks to the App as deployed on the backend. To test a preamble change, deploy the change first, then validate it here.
</Warning>

## Switch models inside a session

Compare how different models handle the same prompt without starting over.

<Steps>
  <Step title="List the models the backend offers">
    ```bash
    aomi model list
    ```
  </Step>
  <Step title="Set the active model for the session">
    ```bash
    aomi model set gpt-5
    ```

    The choice is saved in your local session state, so the next `aomi chat` uses it.
  </Step>
  <Step title="Or apply a model for a single message">
    ```bash
    aomi chat "summarize my portfolio risk" --model claude-sonnet-4
    ```

    Inside the REPL you can do the same with `/model <rig>`, and check the current one with `/model show`.
  </Step>
</Steps>

## Simulate a transaction before you sign

When a chat asks the agent to act on chain, the backend builds the transaction and the CLI queues it locally. You can simulate the queued transaction before signing anything, so no funds move until you are happy with the result.

<Steps>
  <Step title="Trigger a transaction from a chat">
    ```bash
    aomi chat "swap 1 ETH for USDC on Uniswap" \
      --public-key 0xYourAddress \
      --chain 1
    ```

    The CLI prints a queued request, for example `tx-1`.
  </Step>
  <Step title="See what is pending">
    ```bash
    aomi tx list
    ```
  </Step>
  <Step title="Simulate the pending transaction">
    ```bash
    aomi tx simulate tx-1
    ```

    The CLI runs the transaction as an atomic batch against an Anvil snapshot and prints a per step result: success or revert, gas used, value, and a service fee summary. If a step reverts, you see the revert reason. Pass several IDs to simulate them together as one batch.
  </Step>
  <Step title="Sign only if the simulation looks right">
    ```bash
    aomi tx sign tx-1 --private-key 0xYourPrivateKey
    ```

    By default `aomi tx sign` tries account abstraction first and falls back to a direct EOA signature if AA is unavailable. Use `--aa` to require AA, or `--eoa` to force a plain EOA signature.
  </Step>
</Steps>

<Note>
  `aomi tx simulate` works on transactions that are already queued. Run a chat that produces a transaction first, then `aomi tx list` to get the ID.
</Note>

## Debug a tool call

When the model is not calling a tool you expect, run the chat in verbose mode:

```bash
aomi chat "what is the price of ETH?" --verbose
```

Verbose mode streams the tool calls, their arguments, and the raw results live as the agent works. Use it to spot an ambiguous tool description or a missing parameter. Without `--verbose`, only the final reply is printed.

To review the whole exchange after the fact, including every tool result:

```bash
aomi session log
```

## Inspect or resume a session

Each conversation is a session. The CLI keeps a list of them locally so you can pick one back up later.

<Steps>
  <Step title="List your local sessions">
    ```bash
    aomi session list
    ```

    Each session shows an ID and a short label like `session-1`.
  </Step>
  <Step title="Check the active session">
    ```bash
    aomi session status
    ```

    This shows the current session, the model, the chain, and the wallet address in use.
  </Step>
  <Step title="Resume an earlier session">
    ```bash
    aomi session resume session-1
    ```

    The next `aomi chat` continues that conversation. To start clean instead, run `aomi session new`, or add `--new-session` to any chat command.
  </Step>
  <Step title="Read or close a session">
    ```bash
    aomi session log      # full transcript with tool results
    aomi session events   # system events for the session
    aomi session close    # clear the active session pointer
    ```
  </Step>
</Steps>

<Note>
  There is no replay command. To compare old behavior against new behavior after a preamble or tool change, resume the session, read its log with `aomi session log`, then send the same prompt again in a fresh session with `aomi session new`.
</Note>

## Where things live

The CLI stores local state under `~/.aomi` by default. You can override the location with the `AOMI_STATE_DIR` environment variable.

| What | Where |
| --- | --- |
| Active session pointer and client identity | state root under `~/.aomi` |
| State for each session (model, chain, wallet, pending and signed txs) | `~/.aomi/sessions/` |
