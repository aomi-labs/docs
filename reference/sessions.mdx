title: Sessions

Sessions represent individual conversations between a user and an Aomi App. Each session maintains full message history, thread state, and wallet bindings.

## Session Lifecycle

### Three-Tier Load Strategy

1. **Memory cache** — active sessions are held in memory for fast access
2. **Database** — persisted sessions are loaded from the database
3. **Create new** — if no existing session matches, a new one is created

### Message Persistence Schema

Messages are stored with:
- Role (user, assistant, tool)
- Content (text, tool calls, tool results)
- Timestamps
- Metadata (model used, tokens, latency)

## Wallet Binding

Each session can be bound to a wallet address, enabling the agent to transact on behalf of the user within that session.

## Thread Management

- **Create** — start a new thread within a session
- **Switch** — between threads without losing context
- **Rename** — auto-generated titles, editable by the user
- **Delete** — remove threads and their history

## Real-Time Updates

Session state updates are pushed via SSE (`/api/sessions/:id/events`) for real-time UI synchronization.
